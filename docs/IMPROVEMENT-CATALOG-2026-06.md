# Improvement Catalog — 2026-06 (deep audit)

> Generated by a read-only multi-agent deep audit (2026 best-practice research + source scan). Tier A = safe quick win, B = hardening, C = structural/decision-needed.

**Stack:** TypeScript (ESM, Node >=22.13), commander 15 CLI, execa, better-sqlite3, chalk/ora; raw node:http/https webhook server; vitest 4; eslint 10. Integrates Zellij/tmux, git worktree, n8n (GitHub webhook), Obsidian Local REST API, ccusage, optional LiteLLM/Ollama and bubblewrap sandbox.

**Summary:** ccmux is a well-structured CLI with genuinely thoughtful reliability scaffolding (atomic sessions.json lock with stale detection, SQLite INSERT-OR-IGNORE dedup queue, constant-time HMAC, TLS-verified Obsidian calls, array-arg execa avoiding shell injection, a 3-OS CI matrix). The biggest systemic risk is an architectural seam: command functions (newCommand/closeCommand/autoCommand) call process.exit(1) on error, but the same functions are invoked in-process by the long-lived `serve` daemon — so any command failure kills the whole webhook server and makes the queue-rollback / 500-response paths in n8n.ts dead code. Secondary risks cluster in the HTTP server (unbounded request body buffering = CWE-770 DoS, unvalidated session `name` enabling path traversal and feeding attacker-controlled GitHub issue text straight into an autonomous `claude --dangerously-skip-permissions` run) and a permanent dedup-block when an autonomous spawn fails. Test coverage is good for pure functions (HMAC, queue, hooks) but absent for the actual HTTP request handler, the auto/new/close command flows, and worktree create/delete. Most Tier-A fixes are small and behavior-preserving.

This PR implements: SEC-01, SEC-02, SEC-05, DX-01, BUG-02. Deferred: REL-01 + REL-03 → PR #39; REL-05 → Windows lock-race (see notes).

## Tier A

### REL-01 — Command functions call process.exit(1) but are reused in-process by the serve daemon — a single failure kills the whole webhook server ↪ (addressed by open PR #39 (service-layer refactor))
- **severity:** critical | **category:** reliability | **effort:** M | **risk:** medium
- **root cause:** newCommand/closeCommand/autoCommand handle their own errors with `spinner.fail(...); process.exit(1)` (auto.ts:200, new.ts:22/67, close.ts:56/83/165). These are CLI-ergonomic, but n8n.ts calls the SAME functions inside the long-lived HTTP handler. When e.g. createWorktree throws (name collision, dirty tree, git error), the process exits instead of throwing, so the surrounding try/catch in handle() and the `releaseSession()` rollback (n8n.ts:178-184) never execute — they are dead code. One bad webhook drops the entire `serve` process.
- **fix:** Separate concerns: make core command functions THROW on error and never call process.exit; move the `process.exit(1)` to a thin CLI wrapper in index.ts action handlers (catch → print → exit). Then the server's try/catch and queue rollback work as intended. Add a test that drives the handler with a failing newCommand and asserts a 500 (not a process exit).
- **files:** `src/commands/auto.ts`, `src/commands/new.ts`, `src/commands/close.ts`, `src/integrations/n8n.ts`, `src/index.ts`

### SEC-01 — Unbounded request body buffering in the webhook server (CWE-770 memory-exhaustion DoS) ✅ (this PR)
- **severity:** high | **category:** security | **effort:** S | **risk:** low
- **root cause:** readBody() in n8n.ts:19-34 concatenates all `data` chunks into a Buffer with no byte cap and no Content-Length check, then JSON.parse()s it. An unauthenticated POST to /webhook/github (or /session/* when authToken unset) with a multi-GB or many-small-objects body exhausts the heap and crashes the daemon — exactly the Astro GHSA-jm64-8m5q-4qh8 / nodebestpractices pattern. JSON.parse amplifies wire→heap ~15x.
- **fix:** Enforce a limit while reading: track running byte total, and if it exceeds e.g. 1MB destroy the socket and reject with 413. Also reject when Content-Length header exceeds the cap before reading. raw-body-style guard, or a hand-rolled counter in the `data` handler.
- **files:** `src/integrations/n8n.ts`

### SEC-02 — Session `name` is never validated — path traversal into arbitrary worktree directories ✅ (this PR)
- **severity:** high | **category:** security | **effort:** S | **risk:** low
- **root cause:** createWorktree does `path.join(worktreeBase, name)` (worktree.ts:47) and deleteWorktree the same (worktree.ts:134) with no validation of `name`. Via POST /session/new the `name` is fully attacker-controlled when n8n.authToken is unset (default). `name="../../../tmp/x"` resolves the worktree outside worktreeBase; on close/prune the matching `git worktree remove --force` + `fs.rm(wtPath, {recursive,force})` then operates on that escaped path. The CLI `new`/`auto` accept the same unchecked names.
- **fix:** Add a `validateSessionName(name)` guard (allow `^[A-Za-z0-9._/-]+$`, reject leading `-`, reject `..` path segments, reject absolute paths) called at the top of newCommand/autoCommand and both /session/* handlers; reject with a clear error / 400. Pair with `--` before positional git args (see SEC-03).
- **files:** `src/core/worktree.ts`, `src/commands/new.ts`, `src/commands/auto.ts`, `src/integrations/n8n.ts`

### REL-03 — autoCommand leaks its lock on failure (acquireLock without releaseLock) ↪ (addressed by open PR #39 (lock handling in the service layer))
- **severity:** medium | **category:** reliability | **effort:** S | **risk:** low
- **root cause:** auto.ts:61 acquires the per-session lock but the catch block (auto.ts:198) only does spinner.fail + process.exit — it never releaseLock()s. newCommand correctly releases on failure (new.ts:65). Because the lock file stores the PID and acquireLock treats a dead PID as stale, a CLI re-run self-heals; but a same-PID retry or the in-process serve path leaves the lock held, blocking legitimate re-creation with 'already running'.
- **fix:** Wrap autoCommand body so the catch path calls `await releaseLock(sessionName).catch(()=>{})` before rethrowing (mirror new.ts). Best combined with REL-01 (throw, let CLI wrapper exit).
- **files:** `src/commands/auto.ts`, `src/core/lock.ts`

### BUG-01 — listWorktrees matches branches by substring, wrongly claiming non-ccmux branches
- **severity:** medium | **category:** bug | **effort:** S | **risk:** low
- **root cause:** worktree.ts:195 filters with `branch.includes(BRANCH_PREFIX)` where BRANCH_PREFIX='ccmux'. Any branch containing the substring (e.g. `feature/ccmux-docs`, `release/v-ccmux`) is treated as a ccmux worktree, and createWorktree's existence check (worktree.ts:53) and any cleanup logic will operate on unrelated user branches. The subsequent `branch.replace('refs/heads/ccmux/','')` also mis-derives the name for such branches.
- **fix:** Match the prefix anchored: `branch.startsWith('refs/heads/' + BRANCH_PREFIX + '/')` (and derive name only from that). Add a test with a decoy branch like `wip/ccmux-x`.
- **files:** `src/core/worktree.ts`

### DX-01 — Hardcoded personal username "Rikuto" as a fallback path in published tool ✅ (this PR)
- **severity:** medium | **category:** dx | **effort:** S | **risk:** low
- **root cause:** cost.ts:31 `const winUser = process.env.WINDOWS_USERNAME ?? process.env.USER ?? "Rikuto";` then builds `/mnt/c/Users/${winUser}/.claude`. On any other user's WSL2 machine where neither env var is set, ccusage is invoked against a non-existent `/mnt/c/Users/Rikuto/.claude` and silently returns null costs — and it leaks the author's identity into a public repo.
- **fix:** Drop the literal fallback; if no username resolves, skip the WSL2 candidate path entirely (return process.env.CLAUDE_CONFIG_DIR). Resolve the Windows user from `os.userInfo()` / `process.env.USERNAME` rather than a constant.
- **files:** `src/core/cost.ts`

### BUG-02 — reflect --apply claims to deduplicate but always blindly appends ✅ (this PR)
- **severity:** low | **category:** bug | **effort:** S | **risk:** low
- **root cause:** reflect.ts:132 comment says "Deduplicate: don't append if a '## Learned Rules' section already exists at end" but the code unconditionally writes `existing + separator + reflection` (reflect.ts:133-135). Repeated `ccmux reflect <name> --apply` runs append the auto-generated block every time, growing CLAUDE.md without bound.
- **fix:** Either implement the documented check (skip/replace when the auto-generated marker block already exists at EOF) or fix the misleading comment. Replacing the last marked block is the more useful behavior.
- **files:** `src/commands/reflect.ts`

### SEC-05 — Bearer auth comparison is not constant-time ✅ (this PR)
- **severity:** low | **category:** security | **effort:** S | **risk:** low
- **root cause:** checkAuth (n8n.ts:42-47) compares `header === "Bearer " + authToken` with `===`, which short-circuits on first differing byte — a timing side-channel on the static auth token (contrast: the webhook HMAC path correctly uses timingSafeEqual). Token is high-entropy so practical exploitability is low, but it is inconsistent with the project's own HMAC hardening.
- **fix:** Compare with crypto.timingSafeEqual over equal-length buffers (length-guard first), reusing the same pattern as verifyGitHubSignature.
- **files:** `src/integrations/n8n.ts`

### BUG-04 — Obsidian default exchangeRate diverges between modules (150 vs config 155)
- **severity:** low | **category:** bug | **effort:** S | **risk:** low
- **root cause:** obsidian.ts buildDefaultContent/applyTemplate fall back to `data.exchangeRate ?? 150` (obsidian.ts:93,130) while the config DEFAULTS.cost.exchangeRate is 155 (schema.ts:84) and close.ts passes cfg.cost.exchangeRate. If a caller omits exchangeRate the handoff shows a JPY figure off by ~3% from everywhere else, an inconsistency that will confuse cost reconciliation.
- **fix:** Use a single shared default constant for exchangeRate (or make it required in HandoffData so the omission is a type error). Align the literal to 155.
- **files:** `src/integrations/obsidian.ts`, `src/config/schema.ts`

## Tier B

### SEC-03 — git worktree/branch commands omit `--` option terminator before positional args
- **severity:** medium | **category:** security | **effort:** S | **risk:** low
- **root cause:** execa array args prevent shell injection, but git itself still parses a leading-dash positional as an option. `git worktree add -b ccmux/${name} ${wtPath}` (worktree.ts:59) prefixes name with `ccmux/` (safe), but `wtPath` and the no-`-b` fallback `git worktree add ${wtPath} ${branch}` (worktree.ts:66) and `branch -d ${branch}` pass values that, after a traversal-crafted base or future refactor, could begin with `-`. This is the Composer CVE-2024-35241/35242 class.
- **fix:** Insert `--` before positional path/branch args in every git invocation that takes user-derived values: `worktree add -b <branch> -- <path>`, `worktree remove -- <path> --force` (or place --force first), `branch -d -- <branch>`. Combined with SEC-02 validation this fully closes the argument-injection surface.
- **files:** `src/core/worktree.ts`, `src/commands/prune.ts`, `src/commands/merge.ts`

### REL-02 — Failed autonomous spawn permanently blocks re-triggering that issue (dedup row never released due to process.exit)
- **severity:** high | **category:** reliability | **effort:** M | **risk:** medium
- **root cause:** claimSession inserts the dedup row (n8n.ts:166) BEFORE autoCommand runs. The rollback releaseSession (n8n.ts:181) is meant to drop the row on failure, but because autoCommand process.exit(1)s (see REL-01) the catch never runs. Even independent of REL-01: the row is written before the worktree exists, so any crash between claim and a successful close leaves a 'claimed but never completed' row that blocks every future redelivery of issue-N until queue.db is hand-edited.
- **fix:** After fixing REL-01 (throw instead of exit), the existing releaseSession rollback works. Additionally claim AFTER the worktree/session are created, or add a `claimed_at` staleness sweep so rows with no completed_at older than N minutes are reclaimable. Add a test for the claim→fail→release cycle.
- **files:** `src/integrations/n8n.ts`, `src/core/queue.ts`

### TEST-01 — No tests for the actual HTTP request handler (auth, routing, dedup, error mapping)
- **severity:** high | **category:** test | **effort:** M | **risk:** low
- **root cause:** n8n-hmac.test.ts only exercises the pure verifyGitHubSignature function; the integration suite only tests the mock LLM stub. The `handle()` dispatcher — Bearer enforcement, 401 paths, /session/new→newCommand, webhook→claim→auto, 500 rollback, body-size behavior — has zero coverage, which is exactly why REL-01/REL-02 (process.exit kills server; rollback dead code) went unnoticed.
- **fix:** Add a supertest-style suite that starts startServer() on an ephemeral port and asserts: 401 without auth, 200 dedup on duplicate webhook, 400 on missing name, 500 (not process exit) on a failing command. This pins the REL-01 fix.
- **files:** `src/integrations/n8n.ts`, `tests/integration/mock-llm-server.test.ts`

### TEST-02 — Worktree create/delete, lock contention, and command flows are untested
- **severity:** medium | **category:** test | **effort:** M | **risk:** low
- **root cause:** tests cover config merge, queue, hooks scripts, cost, dashboard — but not createWorktree/deleteWorktree against a temp git repo, acquireLock stale-PID recovery, or the new/auto/close happy+failure paths. The Windows-specific retry/prune logic in deleteWorktree (worktree.ts:156-177) is entirely unverified.
- **fix:** Add a temp-repo fixture (`git init` in a tmpdir) and test create→list→delete, dirty-tree guard, force path, and the BUG-01 decoy-branch case. Add a lock test for the dead-PID stale-takeover branch.
- **files:** `src/core/worktree.ts`, `src/core/lock.ts`, `tests/worktree-base.test.ts`

### REL-04 — Server has no request timeout / keep-alive cap — slowloris exposure
- **severity:** medium | **category:** reliability | **effort:** S | **risk:** low
- **root cause:** startServer (n8n.ts:216-231) creates the http/https server with no `server.requestTimeout`, `headersTimeout`, or `keepAliveTimeout`. Combined with the unbounded readBody (SEC-01), a slow client can hold connections open indefinitely. Even bound to 127.0.0.1, n8n/other local processes (or container port-forwards) reach it.
- **fix:** Set server.requestTimeout (e.g. 30s), headersTimeout, and a maxConnections cap after createServer. Cheap defense-in-depth alongside SEC-01.
- **files:** `src/integrations/n8n.ts`

### REL-05 — writeDB on sessions.json is not crash-atomic across the temp file (mode set only on tmp, no fsync) and the lock TTL can race ↪ (deferred — fsync shifts timing and exposes a Windows lock race in the concurrent-create test; needs a lock-layer fix first)
- **severity:** low | **category:** reliability | **effort:** M | **risk:** low
- **root cause:** writeDB (session.ts:105-110) writes a `.tmp` then renames — good — but does not fsync before rename, so a power-loss can leave a zero-length sessions.json on some filesystems. Separately, the file-lock uses a 30s stale TTL based on mtime (session.ts:50); a long withSessionLock op (slow Obsidian/git) under contention could have its lock declared stale and stolen by another process, causing a lost update. Low likelihood given operation duration but worth noting.
- **fix:** fsync the tmp file handle before rename for the durability half. For the lock, refresh mtime during long ops or raise the TTL; the risk is small so this is optional hardening.
- **files:** `src/core/session.ts`

### PERF-01 — Queue prepares SQL statements on every call instead of caching them
- **severity:** low | **category:** perf | **effort:** S | **risk:** low
- **root cause:** claimSession/completeSession/releaseSession call `db().prepare(...)` on every invocation (queue.ts:82-110). better-sqlite3's documented best practice is to prepare once and reuse; under webhook bursts this re-compiles the statement each time. claimSession also calls db() twice. Minor at current volume but trivially avoidable.
- **fix:** Lazily prepare and cache the three statements on the Database instance (e.g. a WeakMap or module-level holder keyed to the open db), invalidated in _closeDbForTests.
- **files:** `src/core/queue.ts`

### REL-06 — prune of orphaned sessions uses non-force delete, which re-blocks on the dead session's uncommitted changes
- **severity:** low | **category:** reliability | **effort:** S | **risk:** low
- **root cause:** pruneCommand without --force calls deleteWorktree (prune.ts:49) which runs the uncommitted-changes guard (worktree.ts:138). An orphaned session is by definition one whose process died — it very likely left uncommitted work, so the default prune throws 'has uncommitted changes' and skips it (prune.ts:55), leaving the orphan stranded forever unless the user knows to pass --force. The UX implies prune cleans orphans but the default rarely does.
- **fix:** For status==='orphaned' specifically, either default to a safe stash/branch-preserving removal or surface a clear 'N orphans skipped — re-run with --force or recover changes at <path>' summary so the dead-end is visible.
- **files:** `src/commands/prune.ts`, `src/core/worktree.ts`

### DX-02 — saveConfig writes mode 0o600 but does not chmod existing files; secrets (authToken/webhookSecret/apiKey) live in a possibly world-readable config
- **severity:** low | **category:** security | **effort:** S | **risk:** low
- **root cause:** config.json holds n8n.authToken, n8n.webhookSecret, obsidian.apiKey, autoclaw.authToken. saveConfig sets mode 0o600 only on create (schema.ts:115); if the file pre-exists with looser perms (created by an editor, copied, or restored from backup) the mode is not re-applied, and on Windows the mode flag is largely ignored. No startup warning if perms are loose.
- **fix:** fs.chmod the config to 0o600 after every save (not just rely on the write mode), and/or doctor-check + warn when config perms are group/other-readable on POSIX. Document that secrets live there.
- **files:** `src/config/schema.ts`, `src/commands/doctor.ts`

### REL-07 — Daemon (non-mux) auto spawn records child.pid but the loop-daemon path does not, so orphan pruning can't track loop sessions
- **severity:** low | **category:** reliability | **effort:** S | **risk:** low
- **root cause:** In the single-shot daemon branch the spawned child's pid is stored via updateSession({pid}) (auto.ts:166), enabling pruneOrphanedSessions to detect death. The loop-daemon branch (spawnLoopDaemon) spawns `bash .ccmux-loop.sh` detached but never returns or records its pid, so the session has no pid and pruneOrphanedSessions (session.ts:179 `if (!session.pid) continue`) skips it forever — a dead loop session is never marked orphaned.
- **fix:** Have spawnLoopDaemon return child.pid and updateSession({status:'busy', pid}) in the loop branch, mirroring the single-shot path.
- **files:** `src/commands/auto.ts`

### DX-03 — verifyGitHubSignature is exported and tested but the event/issue checks run before signature verification, allowing unauthenticated probing of routing logic
- **severity:** low | **category:** security | **effort:** S | **risk:** low
- **root cause:** In handle() the `x-github-event !== 'issues'` early-return (n8n.ts:133) executes BEFORE the HMAC gate (n8n.ts:143), so an unauthenticated caller can probe which event types the server accepts and observe distinct responses without a valid signature. Minor info disclosure; the body is still not acted on, but the ordering means signature verification is not the first gate.
- **fix:** Verify the HMAC signature first (immediately after readBody) and only then branch on event/action. Keeps all accept/reject decisions behind authentication.
- **files:** `src/integrations/n8n.ts`

## Tier C

### SEC-04 — Attacker-controlled GitHub issue body is fed verbatim into an autonomous `claude --dangerously-skip-permissions` run
- **severity:** high | **category:** security | **effort:** L | **risk:** medium
- **root cause:** n8n.ts:160 builds `prompt = Issue #...: ${title}\n\n${issue.body}` from untrusted webhook content and passes it to autoCommand, which launches Claude with --dangerously-skip-permissions (auto.ts:96,148). The only barrier is the regex PreToolUse blocklist (hooks.ts:336) which is easily bypassed (e.g. `r''m -rf`, base64, python -c, env-var indirection) and the write-boundary check. An issue body is a classic indirect prompt-injection vector driving an unsandboxed agent.
- **fix:** Treat webhook-triggered autonomous runs as untrusted: require the bubblewrap `sandbox` path (or refuse on non-Linux), default the webhook lane to a no-network sandbox, and/or gate webhook-driven auto behind an explicit allowlist of repos/authors. At minimum document the trust model and make `--sandbox` the default for the webhook entrypoint. Note the blocklist is defense-in-depth, not a boundary.
- **files:** `src/integrations/n8n.ts`, `src/commands/auto.ts`, `src/core/hooks.ts`

### SEC-06 — Loop daemon writes a generated bash script into the worktree and execs it; relies solely on escapeBashDQ for safety
- **severity:** medium | **category:** security | **effort:** M | **risk:** medium
- **root cause:** spawnLoopDaemon (auto.ts:217-263) string-builds `.ccmux-loop.sh` interpolating maxIter, until-pattern, logFile, and the claude invocation. escapeBashDQ (auto.ts:281) only escapes \ $ " ` for double-quoted contexts, but `MAX_ITER="${maxIter}"` interpolates a value parsed by commander parseInt (numeric, ok) while `UNTIL_PATTERN` comes from user `--until` and logFile from a session-name-derived path. The mitigation is correct for dquote strings but the safety depends on every interpolation staying inside double quotes and on name validation (absent, SEC-02).
- **fix:** After adding SEC-02 name validation, also assert `--until` and derived paths contain no newline; consider passing these via env vars to the script (export then reference) instead of literal interpolation, eliminating the quoting dependency entirely.
- **files:** `src/commands/auto.ts`

### BUG-03 — Stop-hook ccusage cost capture pipes the same stdin twice (already consumed)
- **severity:** low | **category:** bug | **effort:** S | **risk:** low
- **root cause:** In writeStopHook the script reads `INPUT=$(cat)` once (hooks.ts:79), then later re-pipes `echo "$INPUT" | node -e '... process.stdin ...'` to extract session_id (hooks.ts:113-119). That works, but the node snippet reads process.stdin which is the echo pipe — fine — however the surrounding grep checks also re-`echo "$INPUT"`, and on some sh implementations a very large INPUT exceeds ARG_MAX when later used; more concretely the cost-capture only fires when TASK_STATE_FILE exists AND ccusage is installed AND session_id present, a narrow path that is never tested for correctness of the regex Cost-line replacement.
- **fix:** Add a hooks test (like hooks-ccusage.test.ts already does for parts) that feeds a realistic JSON stdin and asserts the Cost line is written/updated exactly once and the iteration/status parsing handles missing fields. Low severity; mostly a coverage/robustness gap.
- **files:** `src/core/hooks.ts`, `tests/hooks-ccusage.test.ts`

### SEC-07 — PreToolUse write-boundary check is prefix-string based and bypassable via symlink / case / relative normalization gaps
- **severity:** medium | **category:** security | **effort:** M | **risk:** medium
- **root cause:** writePreToolUseHook enforces the worktree write boundary by string-prefix comparison `${ABS_PATH#$WORKTREE} != $ABS_PATH` (hooks.ts:373) with an optional realpath fallback only when realpath exists (hooks.ts:379). On Windows MSYS or hosts without realpath -m, a path like `<worktree>/../sibling/file` is not canonicalized and a sibling dir whose name shares the worktree prefix (`<worktreebase>/sessname-evil`) passes the prefix test. Symlinks inside the worktree pointing outward are not resolved on the no-realpath path.
- **fix:** Always canonicalize (node `path.resolve` via the existing node extract helper, or `realpath -m` and fail-closed if unavailable) and compare against worktree + path separator boundary (`$WORKTREE/`), not a bare prefix. This is the agent containment boundary so it should fail closed.
- **files:** `src/core/hooks.ts`

### REL-08 — loadConfig caches the first result process-wide; saveConfig updates the cache but external edits during a long-lived serve are never picked up
- **severity:** low | **category:** reliability | **effort:** S | **risk:** low
- **root cause:** loadConfig memoizes into module-level _config (schema.ts:88-91) and returns it forever. The `serve` daemon loads config once at startup; if an operator edits ~/.ccmux/config.json (rotates a webhook secret, enables Obsidian) the running server never sees it until restart. Tests work around this by swapping CCMUX_DIR, but production behavior is silently stale.
- **fix:** Document that serve must be restarted on config change, or add a SIGHUP handler / mtime check to invalidate _config. At minimum note the caching contract in serve output.
- **files:** `src/config/schema.ts`, `src/commands/serve.ts`
